Networking, HTTP, DNS, And TLS

For SaaS work, you do not need to become a router engineer. You need to know which layers explain which symptoms, and which knobs are dangerous. The useful stack is: DNS names things, IP routes packets, TCP/QUIC moves bytes, TLS authenticates/encrypts, HTTP carries application semantics.

Cross-links: request trace, auth, security, operations.

The Layers That Matter

flowchart TD
  HTTP["HTTP: methods, headers, status, caching, cookies"]
  TLS["TLS: server identity, encryption, certs, ALPN"]
  TCP["TCP or QUIC: connection, loss, congestion, streams"]
  IP["IP: routing between networks"]
  DNS["DNS: name -> address / edge"]
  HTTP --> TLS --> TCP --> IP
  DNS --> HTTP

Mental shortcut:

  • DNS bug: “I cannot find where to go.”
  • TCP/QUIC bug: “I cannot establish or maintain a path.”
  • TLS bug: “I do not trust who answered, or we cannot agree how to encrypt.”
  • HTTP bug: “The application contract was rejected or failed.”

DNS

DNS resolves a name to records:

  • A / AAAA: IPv4/IPv6 address.
  • CNAME: alias to another name.
  • MX: email routing.
  • TXT: verification, SPF/DKIM/DMARC, miscellaneous metadata.

Practical gotchas:

  • TTL controls how long resolvers may cache. Low TTL helps migration but increases query volume.
  • DNS changes are not instant because caches exist at many layers.
  • A CNAME chain can hide where traffic really enters: CDN, load balancer, or vendor.
  • Split-horizon DNS can return different answers inside a VPC than from the public internet.

TCP, QUIC, And Timeouts

TCP gives reliable ordered bytes. QUIC gives reliable encrypted streams over UDP and avoids some TCP head-of-line limitations.

For app engineers, the important knobs are:

  • connection timeout: how long to wait to connect.
  • request timeout: how long the whole operation may take.
  • idle timeout: how long a connection can sit unused.
  • keepalive: whether connections are reused.
  • max connections: resource cap.
  • retry policy: whether to try again after failure.

Failure mode: a client has a 60 second timeout, a load balancer has a 30 second idle timeout, and the app holds the request for 45 seconds. The client sees a mysterious reset or 502, not the app’s intended response.

TLS

TLS provides:

  • server authentication via certificates.
  • encryption in transit.
  • integrity: traffic cannot be modified without detection.
  • optional client authentication via mTLS.

TLS is usually terminated at:

  • CDN/edge.
  • load balancer.
  • ingress gateway.
  • service mesh proxy.
  • the app itself.

Each termination point is a trust boundary. After TLS terminates, traffic may be re-encrypted or travel inside a private network. Do not casually say “TLS means secure” without knowing where it terminates.

HTTP Semantics

Methods:

  • GET: safe read. Should not change state.
  • POST: create/action, not necessarily idempotent.
  • PUT: replace resource, usually idempotent.
  • PATCH: partial update.
  • DELETE: delete, should be idempotent where possible.

Status codes:

  • 2xx: success.
  • 3xx: redirect/cache/navigation.
  • 400: client contract wrong.
  • 401: unauthenticated.
  • 403: authenticated but not allowed.
  • 404: not found or intentionally hidden.
  • 409: conflict with current state.
  • 429: rate limited.
  • 5xx: server or dependency failure.

CORS And Same-Origin Policy

Same-origin policy says browser JavaScript can read responses only from allowed origins. CORS is the server’s controlled exception.

Origin is:

So these are different origins:

CORS is not authentication. It is a browser read-permission mechanism. A server still needs authentication and authorization.

Common CORS confusion:

  • Browser shows CORS error, but the real server returned 500 without CORS headers.
  • Cookies are not sent unless the fetch uses credentials and the server allows credentials.
  • Access-Control-Allow-Origin: * cannot be used with credentials.
  • Preflight OPTIONS must be handled by the server/proxy.

Debugging Checklist

Read from the bottom up:

  1. Did DNS resolve?
  2. Did TCP/QUIC connect?
  3. Did TLS verify?
  4. Did HTTP route?
  5. Did auth pass?
  6. Did the app and dependencies succeed?
